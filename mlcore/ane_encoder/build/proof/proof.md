## Runtime latency (informational)

CPU_ONLY vs CPU_AND_NE wall-time. **Note**: at this demo model size (2 layers × 128 dim × 64 tokens, ~800 KB on disk) the Python→ObjC `predict()` call overhead (~0.5 ms) dominates the per-call cost. ANE compute itself is microseconds. A meaningful speedup needs either (a) a 150M+ param model — the strategic target, ESM2 — or (b) the Xcode performance report, which measures the compiled program directly rather than going through Python.

```
# Latency benchmark — AneEncoder.mlpackage

warmup=5, iters=50, dim=384

  seq     CPU_ONLY (ms)     CPU_AND_NE (ms)     speedup
   64        4.46 ±0.06          1.88 ±0.24       2.37×
  256       33.53 ±1.17         35.15 ±1.06       0.95×

best speedup at seq=64: 2.37×

interpretation:
  Tiny-model latency is dominated by Python→ObjC predict() overhead
  (~0.5 ms/call). ANE compute itself is microseconds on this model.
  Larger seq probes (or larger DIM, which requires re-conversion)
  show ANE's advantage as the compute/overhead ratio shifts.
  For a 150M-param model (e.g. ESM2) the ratio is ~100×, and
  speedups land in the 5–20× range.

  This benchmark is INDIRECT runtime evidence. The authoritative
  proof is the Xcode Core ML Performance Report, which shows the
  per-op compute unit assignment.
```

## Recipe reminder

The four constraints satisfied here, applied identically:

1. **Precision** — `compute_precision=ct.precision.FLOAT16` in convert.py.
2. **Layout**    — `(B, C, 1, S)`, all linear ops are `nn.Conv2d(in, out, 1)`.
3. **Operators** — split-axis LayerNorm, tanh-GELU, attention rewritten without bmm/einsum, no concat.
4. **Shapes**    — batch+dim+dummy axis static; sequence is `ct.RangeDim(32, 256)`.
