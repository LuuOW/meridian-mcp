    Beyond Static Endpoints: Tool Programs

    Static API endpoints are the representational bottleneck of the agentic web. A new framework, ToolPro (arXiv:2606.19992), proposes shifting the interface from stepwise endpoint sequences to executable, effect-typed tool programs, slashing latency by up to 53% and ensuring retry-safe side effects.

    Paper: arXiv:2606.19992

    As LLM-based agents move into production, they are increasingly tasked with long-horizon workflows involving complex service interactions. However, the prevailing interface—stepwise endpoint sequences—fragments coherent plans into local decisions conditioned on intermediate responses. Each step multiplies network round trips and reasoning overhead, while partial failures trigger brittle retries that can duplicate side effects and corrupt service state.

    In "Beyond Static Endpoints: Tool Programs as an Interface for Flexible Agentic Web Services" (arXiv:2606.19992), Liu et al. introduce ToolPro: a system that makes tool intent first-class by representing workflows as executable tool programs. By delegating interaction logic to a server-side sandbox, ToolPro enables turn reduction and enforces rigorous effect-aware semantics.

    The Latency Model: Why Consolidation Wins

    The efficiency gap between stepwise calling ($T_{\text{step}}$) and program execution ($T_{\text{prog}}$) can be modeled by the number of turns $N$. While stepwise execution pays the round-trip penalty ($T_{\text{rtt}}$) and decision overhead ($T_{\text{dec}}$) at every step, a tool program consolidates these into a one-time build cost ($T_{\text{build}}$):

    $$T_{\text{step}} \approx \sum_{i=1}^{N} (T_{\text{rtt}} + T_{\text{dec}} + T_{\text{api}})$$ $$T_{\text{prog}} \approx T_{\text{build}} + (T_{\text{rtt}} + \sum_{i=1}^{N} T_{\text{api}})$$

    The net benefit $\Delta T$ is realized when the savings from turn reduction exceed the program synthesis and compilation time. ToolPro uses a profile-driven policy to predict this delta and adaptively select the optimal execution mode:

    $$\Delta T = (N-1) \cdot (\overline{T_{\text{rtt}}} + \overline{T_{\text{dec}}}) - \overline{T_{\text{build}}}$$

    Exactly-Once Semantics via Effect-Aware Replay

    A major challenge in programmatic tool use is handling partial failures. If a program fails after committing several state-modifying calls, a naive retry would duplicate those side effects. ToolPro resolves this by introducing effect typing (READ vs. WRITE) and a log-based replay mediator.

    The Replay Discipline:
    • READ calls: Always forwarded to the service, as re-issuing them is state-preserving.
    • WRITE calls: Intercepted and matched against a history log $\mathcal{H}$. If a call with identical arguments $(e, a)$ has already completed, the cached outcome is replayed without re-emitting the call.
    • Fail-Closed Guard: If a repaired program attempts to change the order or arguments of a committed WRITE prefix, replay is disabled, and the system falls back to safe stepwise execution.

    Wasm Sandboxing and Implementation

    To execute untrusted, LLM-generated code safely, ToolPro instantiates a WebAssembly (Wasm) runtime. The Wasm module runs with no ambient authority; its only capability is a host-imported unified stub Call(e, a, eff).

    This architecture provides:

    • Strong Isolation: No filesystem or network access beyond the mediated tool boundary.
    • Typed Enforcement: The runtime verifies that effect annotations at each call site match the service's own specifications.
    • Verifiable Feedback: Precise compiler diagnostics and runtime traces are returned to the agent for bounded in-place repair.
    Performance Impact:
    Evaluated on realistic workflows (Memos, Directus, MinIO), ToolPro reduces end-to-end latency by up to 53.4% and client-side traffic by up to 96.1%. The gains are most pronounced in high-latency network conditions and complex, branch-heavy workflows where turn reduction becomes the dominant factor.

    Conclusion

    ToolPro represents a fundamental shift in how we think about agent–service interfaces. By moving away from reactive, stepwise endpoints and toward explicit, executable representations of intent, we can build an agentic web that is not only faster but also significantly more robust.
