Stellar harvest-forecast: from magnetic noise to a 48-hour prediction

We thought we were building a text-retrieval system. We were building a stellar weather forecaster all along — we just needed to point it at the right kind of light. This is the honest accounting.

The reframe

Meridian-mcp was built as a six-archetype text-retrieval router — planet, moon, trojan, asteroid, comet, irregular — classifying documents by light-like properties (wavelength, phase, polarisation, amplitude). It worked, sort of. Better than nothing on benchmarks. Mostly the metaphor, though.

Then a question reframed the whole thing: what is the metaphor actually for? The answer that survived: solar harvesting. Robots will colonise space before humans, and any robot civilisation runs on starlight. Forecasting stellar regimes is in service of harvest — when to deploy collectors, what aperture, what storage allocation — not in service of nicer-looking retrieval rankings.

That reframe gave us a real loss function: kWh harvested per unit aperture per unit time. It also gave us the project we actually built, which is the one this post is about. The text-retrieval surface stays as a downstream demo. The engine underneath got pointed at NASA SPDF (Parker Solar Probe magnetometer) and STScI MAST (JWST spectra).

Pipeline at a glance

    ① INGEST       PSP CDF + JWST x1dints → HF Dataset luuow/meridian-stellar-cache
    ② FEATURES     rolling 1-h windows → (λ, φ, p, a, τ_c) per window — same five
                   features used in the original orbital classifier
    ③ ARCHETYPES   GMM(k=6) on 621 PSP feature vectors, BIC-selected
    ④ HARVESTER    Kepler-derived heliocentric distance r_AU per window;
                   E_harvest = 0.30 · 1361 / r²  (Spectrolab XTJ Prime PV)
    ⑤ TRAIN        five per-archetype Ridge specialists at multiple horizons
    ⑥ EVALUATE     race three policies on held-out E24 — fixed / persistence / archetype
    ⑦ PROJECT      cross-domain z-projection of JWST spectra onto sun-archetype centroids

Everything runs in GitHub Actions free tier and lands on a public Hugging Face Dataset. There is no local hardware, no GPU, no training-time secret sauce. Five tiny specialists, ~10 KB of weights total.

Result — Gate-2 strict pass at the 48-hour horizon

Specialists were trained on 24/48/72/12-hour-shifted (t, t+Δ) pairs from PSP encounters E20–E23, holding out E24 (~92 test pairs at the chosen horizon). Three policies raced on the holdout, scored in MAE of the drift target log10(E_{t+Δ} / E_t):

  • fixed — predict the average training drift (a constant)
  • persistence — predict zero drift, i.e. E_{t+Δ} = E_t
  • archetype — route to the cluster's specialist using the current 5-vector

The interesting result is the horizon dependence. At short horizons the drift is essentially noise — features at time t carry no information about it, and per-cluster Ridge fits noise per cluster, hurting performance. At long horizons, orbital geometry dominates and archetypes meaningfully predict where the spacecraft will be:

At 48 hours, archetype-routing reduces forecast MAE by 35% in log-drift space and by 38% in linear W/m² space (62 kW/m² vs 100 kW/m² persistence). The picture is regime-dependent — archetype routing helps where it matters (long horizons during perihelion) and hurts where there's no signal to extract (short horizons, drift ≈ noise). Both findings are the same kind of honest.

The feature space — six archetypes are real

Stage-3 GMM on 621 windows (z-scored, BIC-selected k=6, diagonal covariance) finds six clusters that map onto the physical archetype expectations. PCA-2D projection below colours each window by its assigned cluster:

PSP feature space (PCA-2D) — six archetypes occupy distinct regions.

The substantive clusters carry their own physics: cluster 0 (planet — steady, polarised) dominates perihelion windows; cluster 1 (cruise — quiet, low-amplitude) dominates the cruise days; cluster 2 (comet — turbulent, short τ_c) and cluster 3 (moon — mixed) split the mid-perihelion turns; cluster 5 (asteroid — narrow-band, large amplitude) catches sparse high-energy events. Cluster 4 is a singleton — one outlier window — and we skip it in training.

Cross-target — Gate-3 with stellar-type discrimination

The harder question: do the sun-derived archetypes mean anything when projected onto OTHER stars? PSP and JWST measure different physics in different units (Hz vs μm, magnetic field vs photon flux), so direct centroid matching is meaningless. The defensible projection is to z-score each dataset within its own sample statistics and match in z-space — encoding the assumption "where this JWST target sits relative to its peers maps to where the corresponding PSP window sits relative to its peers."

With three targets across stellar types — TRAPPIST-1 (M-dwarf), WASP-39 (G-type), WASP-96 (G/K) — the projection is non-degenerate and stellar-type-aware:

JWST → sun-archetype projection — nine spectra, four archetypes used.

Cross-stellar-type discrimination — M-dwarf vs G/G-K differ in 3 of 4 instrument-orders.

In three of four instrument-order configurations, the M-dwarf and G-type targets land in different sun-archetypes. NIRISS Order 1 collapses both stellar types into the same archetype — the broad bandpass apparently dilutes stellar-type signal. NIRSpec NRS1, NIRISS Order 2, and NIRISS Order 3 each discriminate. With only three stars this is preliminary, not statistical — but the projection is doing something, not nothing.

What we did not measure

The headline above is a forecast accuracy claim. The original design diagram had stage-6 racing kWh delivered by three control policies; we built the forecasters, not the controllers. "Loss in kWh" is the framing goal of the project, but the metric we computed is irradiance-forecast MAE. A controller-loop simulator (apply each forecast through a trivial aperture/storage policy, integrate actually-harvested kWh) is the missing piece — small to add, but we haven't.

Final inference architecture

    PSP magnetometer (B_R, B_T, B_N over 1 h, ~16k samples @ 4.6 Hz)
        │
        ▼
    feature extraction (NumPy/SciPy, <1 ms)
        λ_peak       weighted-mean PSD frequency of |B|         (Hz)
        φ_entropy    normalised Shannon entropy of PSD          (·)
        p_polarize   minimum-variance anisotropy ratio          (·)
        a_amplitude  log10 RMS |B| (detrended)                  (·)
        τ_c          first lag where |ACF(|B|)| < 1/e (FFT)     (s)
        │
        ▼
    archetype router (k=6 GMM nearest centroid, z-scored)
        │
        ▼
    per-archetype Ridge specialist (5 weights + 1 bias each)
        │
        ▼
    drift = log10(E_{t+48h} / E_t)        →  E_{t+48h} = E_t · 10^drift

    Total state at inference: ~10 KB (centroids JSON + 5 specialists JSON).
    Edge-deployable on a microcontroller.

Reproduce

Everything is open. Every artifact in the pipeline lives in one HF Dataset:

  • huggingface.co/datasets/luuow/meridian-stellar-cache — raw PSP/JWST + features + archetypes + harvest + specialists + evaluation + projection
  • github.com/LuuOW/meridian-mcp/tree/main/stellar — every script (pull, features, archetypes, harvester, train_eval, project_jwst, plot_results)
  • github.com/LuuOW/meridian-mcp/tree/main/.github/workflows — six workflow files; workflow_dispatch reruns each stage
  • stellar.ask-meridian.uk — live dashboard with the same plots

The data foundation took one workflow run; features another; archetypes one; harvester one; train+eval one; JWST projection one. Total cloud time across all six: under 30 minutes of GitHub Actions free-tier.

What's next, maybe

Three concrete next moves, in order of effort:

  1. Add the missing controller for a real kWh delta. ~50 lines, makes the headline a real harvest-policy claim instead of a forecast-MAE claim.
  2. Expand JWST target list to 10+ stars across stellar types so the cross-target story is statistical instead of preliminary.
  3. Edge deployment: package the 10 KB inference layer for a robot/probe controller. Real-time archetype routing on a CPU is trivial.

For now, the artifacts are on HF, the code is on GitHub, and the dashboard is live. The metaphor earned its keep.
